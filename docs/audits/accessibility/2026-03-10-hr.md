### Critério 1 — HTML Semântico e Landmarks

**Status: PASS**
**WCAG:** 1.3.1 Info and Relationships (Nível A), 2.4.6 Headings and Labels

**Evidências positivas:**

- O layout principal em `src/app/(dashboard)/layout.tsx` (linha 108) utiliza `<main id="main-content" tabIndex={-1}>`, fornecendo o landmark principal obrigatório.
- A barra de navegação principal em `src/components/layout/navbar.tsx` é envolta por `<nav>`, cumprindo o landmark de navegação.
- As páginas de lista (ex.: `employees/page.tsx`) estruturam o conteúdo com `PageLayout > PageHeader > PageBody`, que renderizam `<div>` com IDs semânticos (`id="page"`, `id="page-header"`, `id="page-body"`).
- O componente `Header` (`src/components/layout/header.tsx`, linha 81) renderiza corretamente `<h1>` para o título principal de cada página.
- Modais utilizam o componente `Dialog` da Radix UI, que renderiza `role="dialog"` e `aria-modal="true"` automaticamente via `DialogPrimitive.Content`.
- Títulos de seções nas páginas de detalhe (ex.: `[id]/page.tsx`) utilizam `<h3>` para subdivisões de cards, mantendo hierarquia coerente com o `<h1>` do `Header`.
- A página de visão geral (`overview/page.tsx`) utiliza `CardTitle` dentro de `Card > CardHeader`, com hierarquia de texto consistente.

**Ressalva menor:** `PageLayout`, `PageHeader` e `PageBody` usam `<div>` ao invés de `<main>`, `<header>` e `<section>`. Como o `<main>` já está definido no layout raiz do grupo de rotas, isso não constitui violação, mas o uso de elementos semânticos complementares melhoraria a robustez para leitores de tela.

### Critério 2 — Rótulos ARIA em Elementos Interativos

**Status: FAIL**
**WCAG:** 4.1.2 Name, Role, Value (Nível A)

**Violações identificadas:**

**2.1 — Botões de ação com apenas ícone sem rótulo acessível**

Em múltiplos modais do módulo RH, os botões de fechar ("X"), editar e navegar contêm exclusivamente um ícone Lucide, sem `aria-label` nem texto visível:

```
src/app/(dashboard)/(modules)/hr/(entities)/employees/src/modals/create-modal.tsx
  Linha 364–376: <Button variant="ghost" size="sm" onClick={handleClose}>
                   <X className="w-4 h-4" />
                 </Button>
```

O botão está envolto em `<TooltipTrigger>`, que fornece dica visual ao hover, mas **tooltips não são equivalentes a `aria-label` para leitores de tela**. Um usuário de NVDA ou VoiceOver ouvirá apenas "botão" sem contexto algum.

O mesmo padrão se repete em:

| Arquivo                                   | Linha(s)         | Botão                |
| ----------------------------------------- | ---------------- | -------------------- |
| `employees/src/modals/view-modal.tsx`     | 103–119, 121–135 | Editar, Fechar       |
| `employees/src/modals/view-modal.tsx`     | 103–119          | Editar               |
| `absences/src/modals/view-modal.tsx`      | 85–119           | Ver Detalhes, Fechar |
| `departments/src/modals/create-modal.tsx` | 184–197, 199–213 | Voltar, Fechar       |

**2.2 — Botão de mostrar/ocultar senha sem rótulo acessível**

```
src/app/(dashboard)/(modules)/hr/(entities)/employees/src/modals/create-modal.tsx
  Linha 728–740: <Button type="button" variant="ghost" size="sm"
                   className="absolute right-1 top-1/2 -translate-y-1/2 h-7 w-7 p-0"
                   onClick={() => setShowPassword(!showPassword)}>
                   {showPassword ? <EyeOff ... /> : <Eye ... />}
                 </Button>
```

O mesmo padrão (sem `aria-label`) está em:

- `employees/[id]/page.tsx` (linha 760–775) — modal de criar usuário
- `employees/[id]/edit/page.tsx` — se houver campo de senha

**2.3 — Switch sem rótulo programático associado**

No modal de criar escala de trabalho (`work-schedules/src/modals/create-modal.tsx`, linhas 167–172), cada `Switch` que habilita/desabilita um dia da semana não possui `id` nem `aria-label`. O texto do dia está num `<span>` adjacente mas sem vinculação programática:

```
<Switch
  checked={d.enabled}
  onCheckedChange={checked => updateDay(day, 'enabled', checked)}
/>
<span className="font-medium w-20 text-sm">{getDayLabel(day)}</span>
```

Um leitor de tela anunciará apenas "switch, não marcado" sem indicar a qual dia se refere. O mesmo padrão está no modal de edição (`work-schedules/src/modals/edit-modal.tsx`).

**2.4 — Botões de ação em tabelas (empresas)**

```
src/app/(dashboard)/(modules)/hr/(entities)/companies/src/components/addresses-edit-tab.tsx
  Linha 209: aria-label="Editar endereço"
  Linha 218: aria-label="Remover endereço"
```

Neste arquivo específico, os botões de ação **possuem** `aria-label` — este é um padrão positivo a ser seguido nos demais módulos.

**Correções sugeridas:**

Para os botões com tooltip:

```tsx
// Antes
<Button variant="ghost" size="sm" onClick={handleClose} className="gap-2">
  <X className="w-4 h-4" />
</Button>

// Depois
<Button variant="ghost" size="sm" onClick={handleClose} className="gap-2"
        aria-label="Fechar">
  <X className="w-4 h-4" aria-hidden="true" />
</Button>
```

Para o botão mostrar/ocultar senha:

```tsx
<Button
  type="button"
  variant="ghost"
  size="sm"
  aria-label={showPassword ? 'Ocultar senha' : 'Mostrar senha'}
  onClick={() => setShowPassword(!showPassword)}
>
  {showPassword ? <EyeOff aria-hidden="true" /> : <Eye aria-hidden="true" />}
</Button>
```

Para os Switches de jornada:

```tsx
<Switch
  id={`day-switch-${day}`}
  checked={d.enabled}
  onCheckedChange={checked => updateDay(day, 'enabled', checked)}
  aria-label={`${getDayLabel(day)} — ${d.enabled ? 'habilitado' : 'desabilitado'}`}
/>
```

### Critério 3 — Navegação por Teclado

**Status: WARN**
**WCAG:** 2.1.1 Keyboard (Nível A), 2.1.2 No Keyboard Trap (Nível A)

**O que funciona corretamente:**

- Os componentes `Dialog` e `Select` da Radix UI implementam trap de foco nativamente. Ao abrir um modal, o foco é confinado ao diálogo e a tecla Escape fecha o modal.
- Todos os `Input`, `Button`, `Select` e `Switch` são focalizáveis via Tab por natureza do HTML ou da implementação Radix.
- O componente `Button` (`src/components/ui/button.tsx`) utiliza a tag `<button>` nativa, garantindo foco e ativação via Enter/Space.
- Os cards de empresa na seleção de empresa (`departments/src/modals/create-modal.tsx`, linha 243) utilizam `<Card onClick={...}>` — este é um problema descrito abaixo.

**Problemas identificados:**

**3.1 — Cards clicáveis não são focalizáveis por teclado**

O componente `<Card>` é uma `<div>` com evento `onClick`, sem `tabIndex`, `role="button"` ou `onKeyDown`. Isso impede usuários de teclado de selecionar uma empresa:

```
src/app/(dashboard)/(modules)/hr/(entities)/departments/src/modals/create-modal.tsx
  Linha 243–264:
  <Card
    key={company.id}
    className="p-4 cursor-pointer hover:bg-accent transition-colors"
    onClick={() => handleSelectCompany(company)}
  >
    ...
  </Card>
```

O mesmo padrão pode existir em outros seletores de listas similares.

**3.2 — O wizard de criação de funcionário não tem navegação por teclado explícita entre etapas**

Os botões "Próximo" e "Voltar" em `employees/src/modals/create-modal.tsx` são `<button>` nativos e respondem ao Enter, o que é correto. Porém, ao avançar de etapa, o foco não é reposicionado para o início da nova etapa — o usuário de teclado precisaria Tab várias vezes para reencontrar o contexto.

**3.3 — EmployeeSelector com acessibilidade desconhecida**

O componente `EmployeeSelector` (`src/components/shared/employee-selector.tsx`) é usado em múltiplos modais de RH (ausências, horas extras, férias, controle de ponto). Não foi possível inspecionar diretamente se ele implementa acessibilidade de combobox (role="combobox", aria-expanded, aria-autocomplete). Se usar um `<input>` simples sem esses atributos, a navegação por teclado será inadequada.

**Correções sugeridas:**

Para cards clicáveis:

```tsx
<Card
  key={company.id}
  role="button"
  tabIndex={0}
  className="p-4 cursor-pointer hover:bg-accent transition-colors"
  onClick={() => handleSelectCompany(company)}
  onKeyDown={e => { if (e.key === 'Enter' || e.key === ' ') handleSelectCompany(company); }}
>
```

Para foco ao avançar etapas no wizard, adicionar uma referência ao container da nova etapa e chamar `.focus()` após a transição.

### Critério 4 — Gerenciamento de Foco em Modais e Diálogos

**Status: WARN**
**WCAG:** 2.4.3 Focus Order (Nível A)

**O que funciona corretamente:**

- O componente `Dialog` da Radix UI (`src/components/ui/dialog.tsx`) gerencia foco automaticamente: abre com foco no primeiro elemento focalizável, confina o Tab dentro do diálogo, e retorna o foco ao elemento disparador ao fechar.
- Escape fecha todos os diálogos via `onOpenChange`.
- Modais com formulários frequentemente utilizam `autoFocus` no primeiro campo relevante (ex.: `create-modal.tsx` de funcionários, linha 527: `autoFocus` no campo "Nome Completo"; `departments/create-modal.tsx` usa `useRef` + `focus()` via `useEffect`).

**Problemas identificados:**

**4.1 — Supressão do botão de fechar nativo da Radix sem substituto acessível**

Vários modais do módulo RH utilizam `className="[&>button]:hidden"` para ocultar o botão de fechar padrão da Radix, substituindo-o por um botão customizado na header do diálogo:

```
src/app/(dashboard)/(modules)/hr/(entities)/employees/src/modals/create-modal.tsx
  Linha 346: <DialogContent className="max-w-2xl max-h-[90vh] overflow-y-auto [&>button]:hidden">
```

O botão customizado substituto é um `<Button>` com ícone `<X>` sem `aria-label` (ver Critério 2). Isso significa que o botão de fechar acessível da Radix é removido e substituído por um não acessível.

Arquivos afetados:

- `employees/src/modals/create-modal.tsx` (linha 346)
- `employees/src/modals/view-modal.tsx` (linha 86)
- `absences/src/modals/view-modal.tsx` (linha 71)
- `absences/src/modals/request-sick-leave-modal.tsx` (linha 86)
- `departments/src/modals/create-modal.tsx` (linha 164)
- `vacations/src/modals/create-modal.tsx` (linha 97)
- `absences/src/modals/reject-modal.tsx` (linha 58)

**4.2 — Ausência de `aria-describedby` (DialogDescription) em vários diálogos**

O componente `DialogDescription` da Radix UI conecta automaticamente `aria-describedby` ao `DialogContent`, descrevendo o propósito do diálogo para leitores de tela. A maioria dos modais do módulo RH omite `<DialogDescription>`, prejudicando o contexto anunciado ao abrir o diálogo. Exceção positiva: `absences/src/modals/reject-modal.tsx` usa corretamente `<DialogDescription>` (linha 61–63).

**Correções sugeridas:**

Adicionar `aria-label` nos botões de fechar customizados (ver Critério 2) e incluir `<DialogDescription>` em todos os modais:

```tsx
<DialogHeader>
  <DialogTitle>Novo Funcionário</DialogTitle>
  <DialogDescription>
    Preencha as informações do funcionário em 3 etapas.
  </DialogDescription>
</DialogHeader>
```

### Critério 5 — Contraste de Cores >= 4.5:1

**Status: WARN**
**WCAG:** 1.4.3 Contrast (Minimum) (Nível AA)

**O que funciona corretamente:**

- O sistema de design define tokens semânticos em `src/app/globals.css`: `--color-foreground: var(--os-slate-800)` (#1e2d3b ≈ slate-800) sobre `--color-background: var(--os-gray-50)` (#f9fafb). Esta combinação supera 4.5:1 com larga margem.
- Texto de botões primários: branco (#ffffff) sobre azul (`--os-blue-500` = #3b82f6) tem razão de ~3.0:1 para texto grande (>=18pt bold), atingindo o mínimo de 3:1 para texto grande. Para texto normal (14px/16px), **pode não atingir 4.5:1** — ver problema abaixo.
- Botões destrutivos: branco sobre vermelho (`--os-red-500` = #ef4444) apresenta razão de ~4.1:1, ligeiramente abaixo do mínimo de 4.5:1 para texto normal.

**Problemas identificados:**

**5.1 — Texto de cor `text-slate-500/50` (opacidade 50%) ilegível**

Em múltiplos modais do módulo RH, subtítulos utilizam classes com opacidade reduzida:

```
src/app/(dashboard)/(modules)/hr/(entities)/employees/src/modals/create-modal.tsx
  Linha 356: <span className="text-xs text-slate-500/50 font-normal">Etapa {step} de 3</span>
```

`text-slate-500/50` resulta em `rgb(100 116 139 / 0.5)` sobre fundo branco, produzindo uma cor equivalente a aproximadamente #c7cdd4 — uma razão de contraste de cerca de **1.5:1**, muito abaixo do mínimo de 4.5:1. O mesmo padrão em `employees/src/modals/view-modal.tsx` (linha 94): `text-xs text-slate-500/50`.

**5.2 — `text-muted-foreground` como cor de texto de placeholder**

A variável `--color-foreground-subtle: var(--os-slate-400)` = #94a3b8 sobre fundo branco (#ffffff) apresenta razão de contraste de aproximadamente **2.9:1**, abaixo de 4.5:1. Esta cor é usada extensivamente como placeholder, legendas e ícones informativos.

Nota: `text-muted-foreground` em placeholders e ícones decorativos é tecnicamente permitido quando o texto não é o único indicador de informação. O problema surge quando `text-muted-foreground` é usado para informação semântica, como o subtítulo de etapa acima.

**5.3 — Texto de erro em `text-red-600` sobre fundo colorido**

Alertas destrutivos usam `text-red-600` sobre fundo `bg-red-50`. A razão de contraste de #dc2626 sobre #fef2f2 é aproximadamente **4.5:1**, na margem mínima aceitável.

**Correções sugeridas:**

Para `text-slate-500/50`, substituir por `text-muted-foreground` (mínimo) ou `text-slate-500` (sem opacidade) para garantir legibilidade:

```tsx
// Antes
<span className="text-xs text-slate-500/50 font-normal">Etapa {step} de 3</span>

// Depois
<span className="text-xs text-muted-foreground font-normal">Etapa {step} de 3</span>
```

Para botões primários, considerar usar `--os-blue-600` (#2563eb) como cor de fundo, que atinge razão de ~4.5:1 com texto branco em tamanho normal.

### Critério 6 — `htmlFor` em Todos os Labels de Formulário

**Status: FAIL**
**WCAG:** 1.3.1 Info and Relationships (Nível A), 3.3.2 Labels or Instructions (Nível A)

**Violações sistemáticas identificadas:**

Múltiplos `<Label>` no módulo RH são utilizados sem `htmlFor`, desvinculando o rótulo do controle de formulário correspondente. Leitores de tela dependem desta vinculação para anunciar o rótulo ao focar o controle.

| Arquivo                                            | Linha                   | Label sem htmlFor                             | Controle afetado                |
| -------------------------------------------------- | ----------------------- | --------------------------------------------- | ------------------------------- |
| `overtime/src/modals/create-modal.tsx`             | 90                      | `<Label>Funcionário *</Label>`                | `EmployeeSelector`              |
| `vacations/src/modals/create-modal.tsx`            | 111                     | `<Label>Funcionário *</Label>`                | `EmployeeSelector`              |
| `absences/src/modals/request-sick-leave-modal.tsx` | 94                      | `<Label>Funcionário *</Label>`                | `EmployeeSelector`              |
| `time-control/src/modals/clock-in-modal.tsx`       | 67–69                   | `<Label>Funcionário *</Label>`                | `EmployeeSelector`              |
| `time-control/src/modals/clock-out-modal.tsx`      | 67                      | `<Label>` sem htmlFor                         | `EmployeeSelector`              |
| `departments/[id]/edit/page.tsx`                   | 303                     | `<Label>Empresa Vinculada</Label>`            | Seletor de empresa              |
| `companies/src/modals/stakeholder-modal.tsx`       | 119                     | `<Label className="mb-1.5">Nome *</Label>`    | `Input` sem id                  |
| `companies/src/modals/stakeholder-modal.tsx`       | 129                     | `<Label className="mb-1.5">Documento</Label>` | `Input` sem id                  |
| `companies/src/modals/stakeholder-modal.tsx`       | 140                     | `<Label className="mb-1.5">Cargo *</Label>`   | `SelectTrigger` sem id          |
| `companies/src/modals/stakeholder-modal.tsx`       | 159, 169, 178, 187, 196 | Todos sem htmlFor                             | Controles sem id correspondente |
| `companies/src/modals/stakeholder-modal.tsx`       | 213                     | `<Label>Representante Legal</Label>`          | `Switch` sem id                 |

O arquivo `companies/src/modals/stakeholder-modal.tsx` é o caso mais crítico, com **8 labels** sem vinculação — todos os campos do formulário de sócio carecem de associação.

**Padrão correto observado em outros arquivos (referência):**

```tsx
// Correto — usado em departments/src/modals/create-modal.tsx
<Label htmlFor="name">Nome do Departamento *</Label>
<Input id="name" ... />
```

**Correções sugeridas:**

Para `EmployeeSelector` (componente compartilhado), adicionar suporte a `id` prop e passar `htmlFor` correspondente:

```tsx
// No modal
<Label htmlFor="employee-selector">Funcionário *</Label>
<EmployeeSelector id="employee-selector" value={...} onChange={...} />
```

Para `stakeholder-modal.tsx`, adicionar `id` único a cada controle e `htmlFor` ao Label correspondente:

```tsx
<Label htmlFor="stakeholder-name">Nome *</Label>
<Input id="stakeholder-name" value={form.name || ''} ... />
```

### Critério 7 — Texto Alternativo em Imagens

**Status: WARN**
**WCAG:** 1.1.1 Non-text Content (Nível A)

**Imagens identificadas no módulo RH:**

**7.1 — Foto de funcionário com `alt` correto**

```
src/app/(dashboard)/(modules)/hr/(entities)/employees/[id]/page.tsx
  Linha 408–412: <img src={photoDisplayUrl} alt={employee.fullName} ... />
  Linha 476–480: <img src={photoDisplayUrl} alt={employee.fullName} ... />
```

Ambas as instâncias utilizam o nome completo do funcionário como texto alternativo, o que é **correto e adequado**.

**7.2 — Ícones decorativos Lucide sem `aria-hidden`**

Os ícones Lucide (ex.: `<Users className="h-5 w-5" />`) dentro de botões e cards informativos **não possuem** `aria-hidden="true"`. Quando um leitor de tela encontra um SVG sem `aria-hidden`, pode tentar ler o conteúdo do SVG, resultando em anúncios ruidosos. Os ícones Lucide renderizam SVGs com `aria-hidden="true"` por padrão a partir de versões recentes, mas isso deve ser confirmado para a versão utilizada.

**7.3 — Ausência de imagem não marcada como decorativa**

No card de estado vazio da página de detalhe (`[id]/page.tsx`, linha 353):

```tsx
<Users className="w-16 h-16 mx-auto mb-4 text-muted-foreground" />
```

Este ícone é decorativo (o texto adjacente já comunica o estado). Seria adequado adicionar `aria-hidden="true"`.

**7.4 — Charts da visão geral sem descrição acessível**

A página `hr/overview/page.tsx` contém gráficos Recharts (BarChart, PieChart, LineChart) sem descrição acessível. O Recharts não adiciona `role="img"` nem `aria-label` por padrão. Usuários de leitores de tela recebem apenas os dados da tabela implícita, sem contexto do tipo ou propósito do gráfico.

**Correções sugeridas:**

Para ícones decorativos grandes:

```tsx
<Users
  className="w-16 h-16 mx-auto mb-4 text-muted-foreground"
  aria-hidden="true"
/>
```

Para gráficos, adicionar um elemento de descrição acessível:

```tsx
<div role="img" aria-label="Gráfico de barras: funcionários por departamento">
  <ResponsiveContainer>
    <BarChart ... />
  </ResponsiveContainer>
</div>
```

### Critério 8 — Link de Pular Navegação

**Status: PASS**
**WCAG:** 2.4.1 Bypass Blocks (Nível A)

**Evidência:**

O layout do grupo de rotas `(dashboard)` em `src/app/(dashboard)/layout.tsx` (linhas 93–98) implementa corretamente o link de pular conteúdo:

```tsx
<a
  href="#main-content"
  className="sr-only focus:not-sr-only focus:absolute focus:top-2 focus:left-2
             focus:z-50 focus:px-4 focus:py-2 focus:bg-white focus:text-blue-600
             focus:rounded"
>
  Pular para o conteudo principal
</a>
```

O link é visualmente oculto (`sr-only`) e se torna visível ao receber foco, direcionando ao elemento `<main id="main-content" tabIndex={-1}>`. O padrão está correto.

**Observação menor:** O texto "Pular para o conteudo principal" carece de acento em "conteúdo". Embora não seja uma violação de acessibilidade técnica, recomenda-se corrigir para "Pular para o conteúdo principal" por consistência com a política de idioma em português formal do projeto.

### Critério 9 — Sem `outline: none` sem Substituto de Foco Visível

**Status: WARN**
**WCAG:** 2.4.7 Focus Visible (Nível AA), 2.4.11 Focus Appearance (Nível AA — 2.2)

**Ocorrências de `outline: none` identificadas:**

**9.1 — Célula de edição da planilha de importação**

```
src/app/globals.css, linha 973–974:
.Spreadsheet__cell input,
.Spreadsheet__data-editor input {
  outline: none !important;
}
```

Este bloco remove o outline do input em edição de células da planilha (`import/hr/`). Contudo, o contexto é a planilha de importação — componente de terceiros — e o foco da célula é indicado pela `box-shadow: inset 0 0 0 2px var(--spreadsheet-selected-border)` aplicada à célula pai (linha 965). Este substituto visual é adequado desde que o contraste da borda seja suficiente.

**9.2 — Editor de texto rico ProseMirror**

```
src/app/globals.css, linha 1652–1654:
.ProseMirror {
  outline: none;
}
```

O editor ProseMirror (usado no módulo de e-mails, mas afeta toda a aplicação) remove o outline. Não foi identificado um substituto de foco visível para o contêiner do editor. Usuários de teclado que focam o editor não recebem indicação visual do foco, exceto se o cursor de texto aparecer naturalmente.

**9.3 — Button base sem `focus-visible`**

O componente `Button` em `src/components/ui/button.tsx` utiliza `outline-none` no CVA base:

```
"inline-flex items-center ... outline-none select-none ..."
```

Não há classes `focus-visible:ring-*` ou `focus-visible:outline-*` no componente base. O Tailwind fornece `focus-visible:ring-2 focus-visible:ring-ring` via plugin de formulários, mas este padrão não está explícito no `Button`. O comportamento real depende de configuração global do Tailwind que pode ou não estar ativa.

O `Input` (`src/components/ui/input.tsx`, linha 17) corretamente implementa:

```
focus:ring-[3px] focus:ring-[rgb(var(--color-ring)/0.5)]
```

Este é o padrão esperado que deve ser replicado no `Button`.

**Correções sugeridas:**

Para `Button`, adicionar foco visível explícito:

```tsx
// No buttonVariants base, adicionar:
'focus-visible:ring-2 focus-visible:ring-[rgb(var(--color-ring))] focus-visible:ring-offset-2';
```

Para ProseMirror, adicionar um substituto de foco:

```css
.ProseMirror {
  outline: none;
}
.ProseMirror:focus {
  box-shadow: 0 0 0 2px rgb(var(--color-ring) / 0.5);
  border-radius: var(--input-radius);
}
```

## Priorização de Correções

### Prioridade 1 — Crítico (implementar imediatamente)

1. **Adicionar `aria-label` a todos os botões ícone** (Fechar, Editar, Mostrar Senha, Voltar) em todos os modais do módulo RH — afeta diretamente usuários de leitores de tela.
2. **Corrigir todos os `<Label>` sem `htmlFor`** em `stakeholder-modal.tsx` e modais com `EmployeeSelector` — vinculação programática obrigatória pelo WCAG 1.3.1.

### Prioridade 2 — Alto (resolver no próximo sprint)

3. **Adicionar `focus-visible:ring-*` ao componente `Button`** para garantir foco visível em toda a aplicação.
4. **Adicionar `<DialogDescription>` aos modais** que não a possuem, melhorando o contexto anunciado por leitores de tela ao abrir diálogos.
5. **Tornar Cards clicáveis acessíveis por teclado** com `role="button"`, `tabIndex={0}` e handler de `onKeyDown`.
6. **Adicionar `aria-label` aos Switches de dias da semana** em escalas de trabalho.

### Prioridade 3 — Médio

7. **Substituir `text-slate-500/50` por cores com contraste adequado** (mínimo `text-muted-foreground`).
8. **Adicionar `aria-hidden="true"` a ícones decorativos grandes** em estados vazios e cards informativos.
9. **Adicionar descrição acessível aos gráficos** da visão geral de RH (`role="img"` + `aria-label`).
10. **Corrigir ortografia** do link de pular navegação: "conteúdo" com acento.

### Prioridade 4 — Baixo

11. **Adicionar substituto de foco ao ProseMirror** (`box-shadow` ao receber `:focus`).
12. **Verificar o componente `EmployeeSelector`** para conformidade com o padrão ARIA de combobox.
13. **Investigar contrastes dos botões primários** com ferramentas de medição precisas (ex.: WebAIM Contrast Checker) usando os tokens de cor definidos.

## Pontos Positivos do Módulo

- Implementação do link "Pular para o conteúdo principal" no layout raiz.
- Uso de componentes Radix UI para Dialog, Select e Switch, que fornecem comportamentos ARIA fundamentais.
- Ícones decorativos nas abas de edição de empresas (`addresses-edit-tab.tsx`, `cnaes-edit-tab.tsx`, `stakeholders-edit-tab.tsx`) possuem `aria-label` adequados nos botões de ação.
- O arquivo `reject-modal.tsx` (ausências) é o único modal do módulo que usa corretamente `<DialogDescription>`.
- O sistema de design em `globals.css` inclui `@media (prefers-reduced-motion: reduce)` desabilitando animações para usuários com sensibilidade a movimento — excelente prática de acessibilidade.
- Os campos de formulário da maioria dos modais usam corretamente `<Label htmlFor="id"><Input id="id">`, com a notável exceção dos arquivos citados no Critério 6.
- A hierarquia de títulos (`h1` no `Header`, `h3` em seções de cards) é consistente na maioria das páginas de detalhe.
