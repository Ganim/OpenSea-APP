## Resumo Executivo

O módulo de RH é construído sobre um sistema de design consistente (OpenSea OS) que fornece boas bases responsivas. No entanto, existem falhas sistêmicas em determinados padrões que afetam a usabilidade em dispositivos móveis, especialmente nas páginas de filtros com larguras fixas, no "Identity Card" das páginas de detalhe e na barra de seleção flutuante (SelectionToolbar) que pode transbordar em telas estreitas.

**Score geral:** 6,5 / 10

## Análise Detalhada por Critério

### Critério 1 — Abordagem Mobile-First

**Resultado: WARN**

A maioria dos componentes compartilhados do layout usa mobile-first corretamente. Exemplos positivos:

- `PageHeroBanner`: `p-8 md:p-12` — correto.
- `PageDashboardSections`: `grid-cols-1 md:grid-cols-2 lg:grid-cols-3` — correto.
- `EntityGrid`: `gridColumns = 'grid-cols-1 md:grid-cols-2 lg:grid-cols-3 xl:grid-cols-4'` — correto.
- `Header` (botões): `hidden sm:inline` para ocultar texto em mobile — correto.
- `TabsTrigger` ícones: `hidden sm:inline` — correto.

Problema encontrado nas páginas de detalhe: o "Identity Card" usa `flex items-start gap-5` sem breakpoint, sem `flex-col` como base. Em telas de 320 px, os três elementos (ícone + título + coluna de datas) competem por espaço horizontal. O padrão correto seria `flex-col sm:flex-row`.

**Arquivo afetado:**

- `src/app/(dashboard)/(modules)/hr/(entities)/employees/[id]/page.tsx` (linha 406)
- `src/app/(dashboard)/(modules)/hr/(entities)/companies/[id]/page.tsx` (linha 257)
- `src/app/(dashboard)/(modules)/hr/(entities)/payroll/[id]/page.tsx` (linha 241)
- Todos os demais `[id]/page.tsx` do módulo seguem o mesmo padrão.

**Correção:**

```tsx
// Antes
<div className="flex items-start gap-5">

// Depois
<div className="flex flex-col sm:flex-row items-start gap-4 sm:gap-5">
```

A coluna de datas à direita deve mover-se para baixo em mobile:

```tsx
// Antes
<div className="flex flex-col gap-2 shrink-0 text-sm">

// Depois
<div className="flex flex-row sm:flex-col gap-4 sm:gap-2 shrink-0 text-sm mt-2 sm:mt-0">
```

### Critério 2 — Sem Overflow Horizontal em Mobile

**Resultado: FAIL (sistêmico)**

Este é o problema mais grave do módulo. Múltiplas páginas de listagem renderizam barras de filtros com larguras fixas dentro de um `div` com `flex flex-wrap`, mas as larguras mínimas individuais excedem 320 px mesmo com wrapping.

**Padrões problemáticos encontrados:**

**a) `w-64` (256 px) para `EmployeeSelector`**

Presente em `/hr/absences` (linha 521), `/hr/overtime`, `/hr/time-control` (linha 161), `/hr/vacations` e outros:

```tsx
<div className="w-64">
  <EmployeeSelector ... />
</div>
```

Em telas de 320 px, 256 px é 80% da largura, deixando apenas 64 px para o padding do container. O conteúdo do seletor pode transbordar.

**b) Filtros com múltiplas larguras fixas em linha**

Em `/hr/absences` (linha 520–583):

```tsx
<div className="flex flex-wrap items-center gap-3">
  <div className="w-64">...</div>        {/* 256px */}
  <SelectTrigger className="w-52">...</SelectTrigger>  {/* 208px */}
  <SelectTrigger className="w-44">...</SelectTrigger>  {/* 176px */}
  <DateRangeFilter ... />                 {/* largura desconhecida, com inputs de data */}
```

A soma ultrapassa 640 px. Embora `flex-wrap` quebre os itens em novas linhas, cada item com largura fixa pode ainda exceder 320 px individualmente ou criar overflow.

**c) Filtros em `/hr/payroll` (linha 497–532):**

```tsx
<SelectTrigger className="w-[160px]">  {/* 160px */}
<Input className="w-[100px]" />        {/* 100px */}
<SelectTrigger className="w-[160px]">  {/* 160px */}
```

Este é o caso menos crítico, pois os valores são menores.

**Correção sistêmica para filtros:**

```tsx
// Substituir larguras fixas por classes responsivas
<div className="w-full sm:w-64">   {/* Substitui: <div className="w-64"> */}

<SelectTrigger className="w-full sm:w-52">   {/* Substitui: className="w-52" */}
```

O container principal dos filtros também deve receber `overflow-x-auto` como fallback de segurança:

```tsx
<div className="flex flex-wrap items-start gap-3 overflow-x-auto">
```

### Critério 3 — Touch Targets Mínimos 44×44 px

**Resultado: WARN**

A maioria dos botões de ação usa `size="default"` (40 px de altura padrão do shadcn/ui) ou `size="sm"` (32 px). Os botões de ação rápida nos cards (Aprovar/Rejeitar em `/hr/absences`, Calcular/Aprovar/Pagar em `/hr/payroll`) usam `size="sm"` com altura de 32 px — abaixo do mínimo recomendado de 44 px para touch.

**Exemplos problemáticos:**

Em `/hr/absences/page.tsx` (linhas 343–372):

```tsx
<Button size="sm" variant="outline" className="flex-1 text-xs ...">
  Aprovar
</Button>
<Button size="sm" variant="outline" className="flex-1 text-xs ...">
  Rejeitar
</Button>
```

Em `/hr/payroll/page.tsx` (linhas 322–366):

```tsx
<Button variant="outline" size="sm" className="flex-1 gap-1 ...">
  Calcular
</Button>
```

**Problema adicional:** O botão de toggle de senha no modal de criação de funcionário usa `h-7 w-7` (28 px), claramente abaixo do mínimo:

```tsx
className = 'absolute right-1 top-1/2 -translate-y-1/2 h-7 w-7 p-0';
```

**Correção:**

- Substituir `size="sm"` por `size="default"` nos botões de ação rápida dentro de cards em mobile, ou aplicar `min-h-[44px] min-w-[44px]` via classe utilitária.
- Aumentar o botão de toggle de senha: `h-9 w-9` (36 px, ainda borderline) ou `h-11 w-11` (44 px).

### Critério 4 — Tabelas Responsivas

**Resultado: PASS**

O módulo de RH não utiliza elementos `<table>` diretamente. Os dados são exibidos via `EntityCard` (modo grid e lista) e via cards de informação com `InfoField`. O `EntityGrid` usa grids CSS nativos que colapsam corretamente com `grid-cols-1` em mobile. As páginas de detalhe usam `grid md:grid-cols-2` ou `grid md:grid-cols-6` que colapsam para uma coluna em mobile.

A única exceção potencial é o `SelectionToolbar` flutuante (ver critério 11), mas não é uma tabela.

### Critério 5 — Navegação Funciona em Mobile

**Resultado: PASS**

O `Navbar` é uma barra fixa flutuante que usa `w-[calc(100%-2rem)]` — adapta-se à largura da tela. O menu é acionado pelo botão hambúrguer (Grid3x3) que abre o `ToolsPanel` como overlay modal em tela cheia (`fixed inset-4`). Em mobile, o `ToolsPanel` usa `grid-cols-2` para o menu de navegação (sem prefixo, mas como valor padrão funciona em mobile) e `inset-4` para margem.

O `pt-28` do `main` no layout do dashboard acumula espaço suficiente para a navbar flutuante em todos os dispositivos.

**Ponto positivo:** O `ToolsPanel` com `max-h-[60vh] overflow-y-auto` garante scroll interno em viewports pequenos.

**Ponto de atenção menor:** O `ToolsPanel` em mobile (320 px) com `p-8` no header pode ser excessivo — considerar reduzir para `p-4 sm:p-8`.

### Critério 6 — Formulários Usáveis em Mobile

**Resultado: WARN**

**Positivo:**

- O `CreateModal` de funcionários usa campos em coluna única dentro do Dialog — correto para mobile.
- Os `Input`, `Select` e `Label` do shadcn/ui são intrinsecamente de largura total dentro de seus containers.
- O `EntityForm` (usado na edição de empresas) provavelmente renderiza campos em 1 coluna em mobile — não foi possível verificar sem ler o código completo do componente.

**Problemas:**

a) O `Header` component usa `flex-row` sem breakpoint:

```tsx
// src/components/layout/header.tsx (linha 72-76)
<div className={cn(
  'flex items-center justify-between',
  'flex-row gap-6',  // sem flex-col em mobile
  className
)}>
```

Em mobile com título longo + botões, o título é comprimido ou truncado. A correção seria `flex-col sm:flex-row` com os botões à direita em `sm:`.

b) O `PageActionBar` usa `span className="hidden lg:inline"` para os textos dos botões (linha 110), o que é adequado para botões compactos, mas o componente não tem um breakpoint entre `sm:` e `lg:` — em tablets (768–1023 px), os botões ficam sem rótulo.

**Correção para `Header`:**

```tsx
<div className={cn(
  'flex flex-col gap-4 sm:flex-row sm:items-center sm:justify-between sm:gap-6',
  className
)}>
```

### Critério 7 — Modais/Diálogos Cabem no Viewport Mobile

**Resultado: WARN**

**Positivo:**

- O `Dialog` do shadcn/ui aplica `max-w-[calc(100%-2rem)]` por padrão em mobile.
- O `CreateModal` de funcionários usa `max-w-2xl max-h-[90vh] overflow-y-auto` — correto.
- O modal de "Criar Usuário" usa `max-w-md` — adequado.

**Problemas:**

a) O `CreateModal` de funcionários contém um scroll vertical (`overflow-y-auto`), mas o conteúdo interno pode ser longo em dispositivos de 320 px, especialmente na Etapa 3 com os campos de usuário e grupos de permissão expandidos. O `[&>button]:hidden` remove o botão X nativo do Dialog e o reposiciona manualmente no header, mas o header do dialog usa `flex flex-row items-center justify-between` sem quebra de linha em mobile.

b) O modal `max-w-2xl` (672 px) em dispositivos de 320 px vai colapsar para `calc(100% - 2rem)` = 288 px pelo comportamento padrão do shadcn/ui — isso está correto. No entanto, o conteúdo interno do step 3 com múltiplos campos expande significativamente.

c) Não há uso de `Sheet` (drawer lateral/bottom sheet) para qualquer modal do módulo — os modais são sempre dialogs centralizados. Em mobile, um bottom sheet seria mais ergonômico para ações como "Registrar Atestado" ou "Registrar Entrada/Saída".

**Correção imediata:** Verificar se o conteúdo interno dos modais multi-step tem padding interno excessivo em mobile. Reduzir `space-y-6` para `space-y-4` dentro de dialogs.

### Critério 8 — Texto Legível sem Zoom

**Resultado: PASS**

- Títulos de página usam `text-3xl` (30 px) ou `text-2xl` (24 px) — adequados.
- Texto de conteúdo usa `text-sm` (14 px) para metadados secundários e `text-base` implícito para o corpo — dentro dos limites aceitáveis.
- O único uso de `text-xs` problemático é nos metadados dos cards de Ausências e Folha de Pagamento (14 exceções), mas são informações secundárias, não o texto principal.
- `text-3xl md:text-4xl` no `PageHeroBanner` segue mobile-first corretamente.

**Ponto de atenção:** `text-xs` para botões de ação dentro de cards (`className="flex-1 text-xs ..."`) pode ser difícil de ler em mobile. Recomenda-se `text-sm` mínimo.

### Critério 9 — Imagens/Mídia Escalam Corretamente

**Resultado: PASS**

- A foto do funcionário usa `className="w-full h-full object-cover aspect-square rounded-lg"` — escala corretamente.
- Os gráficos Recharts usam `ResponsiveContainer width="100%" height={280}` — correto e adaptável.
- Ícones são SVGs inline via Lucide — escalam infinitamente.

**Ponto de atenção nos gráficos:** O `YAxis` do gráfico de barras horizontal na `/hr/overview` usa `width={120}` fixo:

```tsx
<YAxis type="category" dataKey="name" width={120} tick={{ fontSize: 12 }} />
```

Em mobile, 120 px para o eixo Y consome ~37% de um viewport de 320 px, deixando apenas ~200 px para as barras. Em dispositivos menores que 320 px isso seria crítico, mas para 320 px é borderline aceitável.

### Critério 10 — Grids Adaptam Colunas por Breakpoint

**Resultado: WARN**

**Positivo:**

- `EntityGrid` com `gridColumns = 'grid-cols-1 md:grid-cols-2 lg:grid-cols-3 xl:grid-cols-4'` — correto.
- `PageDashboardSections`: `grid-cols-1 md:grid-cols-2 lg:grid-cols-3` — correto.
- KPI cards na `/hr/overview`: `grid-cols-1 sm:grid-cols-2 lg:grid-cols-4` — correto.

**Problemas:**

a) As abas nas páginas de detalhe de empresa (`CompanyDetailPage`, `CompanyEditPage`) usam `TabsList className="grid w-full grid-cols-5 mb-4 p-2 h-12"`. Com 5 colunas e altura fixa de 48 px em um TabsList de largura total, em 320 px cada tab tem apenas 64 px de largura. O texto de cada aba ("Endereços", "Sócios", "CNAEs", "Fiscal", "Documentos") é truncado ou ilegível.

```tsx
// src/app/(dashboard)/(modules)/hr/(entities)/companies/[id]/page.tsx (linha 297)
<TabsList className="grid w-full grid-cols-5 mb-4 p-2 h-12">
```

**Correção:** Usar `grid-cols-2 sm:grid-cols-5` ou mudar para um componente de Tabs scrollável horizontalmente:

```tsx
<TabsList className="flex overflow-x-auto sm:grid sm:grid-cols-5 w-full mb-4 p-2 h-auto sm:h-12">
```

b) A `/hr/employees/[id]` usa `grid md:grid-cols-6` para o bloco de informações pessoais sem colapso explícito em mobile (sem prefixo base), o que funciona mas não é mobile-first. A coluna 1 (foto) e as 5 colunas de campos aparecem em coluna única em mobile pelo comportamento padrão — aceitável, mas a semântica está implícita, não explícita.

### Critério 11 — Alternativa Touch para Drag-and-Drop

**Resultado: WARN**

O `EntityGrid` implementa drag-selection via eventos de mouse (`mousedown`, `mousemove`, `mouseup`). Não há equivalente para eventos de touch (`touchstart`, `touchmove`, `touchend`). Em mobile, a seleção de múltiplos itens não é possível via drag.

```tsx
// src/core/components/entity-grid.tsx (linhas 339–358)
const handleMouseDown = useCallback((e: React.MouseEvent) => {
  // Só ativa drag selection com botão esquerdo
  if (e.button !== 0) return;
  ...
}, [enableDragSelection]);
```

**Mitigação existente:** A seleção individual por toque em um item ainda funciona (via `onClick`), e os botões de ação inline nos cards (Aprovar, Calcular, etc.) permitem ações sem necessidade de multi-seleção. O `SelectionToolbar` com "Selecionar todos" compensa a ausência de drag-select em mobile.

**Problema adicional:** O `SelectionToolbar` flutuante usa `fixed left-1/2 -translate-x-1/2` com conteúdo horizontal sem `flex-wrap`. Em mobile com 4 ações primárias mais o contador, o toolbar pode transbordar horizontalmente em telas de 320 px.

```tsx
// src/core/components/selection-toolbar.tsx (linha 180)
<div className="flex items-center gap-2 px-4">
```

**Correção para o SelectionToolbar:**

```tsx
<div className="flex flex-wrap items-center gap-2 px-3 max-w-[calc(100vw-2rem)]">
```

### Critério 12 — Paginação Acessível em Mobile

**Resultado: PASS**

O módulo de RH não utiliza paginação paginada no frontend — usa carregamento com `perPage: 100` e exibe todos os resultados em grid. Não há componente de paginação com botões de página que possa causar overflow. A rolagem do grid nativo é o mecanismo de navegação.

## Prioridades de Correção

### Prioridade 1 — Crítica (impacto em todas as páginas de listagem)

**Problema:** Barra de filtros com larguras fixas (`w-64`, `w-52`, `w-44`, `w-[160px]`) causa overflow em 320 px.

**Páginas afetadas:** `/hr/employees`, `/hr/absences`, `/hr/overtime`, `/hr/vacations`, `/hr/payroll`, e quaisquer outras com `EmployeeSelector` ou múltiplos `SelectTrigger` em linha.

**Correção:**

1. Substituir `w-64` por `w-full sm:w-64` nos wrappers de `EmployeeSelector`.
2. Substituir `w-52`, `w-44`, `w-[160px]` dos `SelectTrigger` por `w-full sm:w-52` etc.
3. Adicionar `overflow-x-hidden` no `PageBody` ou `overflow-x-auto` no container de filtros como fallback.

---

### Prioridade 2 — Alta (impacto nas páginas de detalhe mais visitadas)

**Problema:** "Identity Card" nas páginas de detalhe usa `flex items-start` sem colapso em mobile. A coluna de datas (criado/atualizado) comprime o título em 320 px.

**Páginas afetadas:** `/hr/employees/[id]`, `/hr/companies/[id]`, `/hr/payroll/[id]`, `/hr/departments/[id]`, `/hr/positions/[id]`, `/hr/work-schedules/[id]`.

**Correção:**

```tsx
// Wrapper do Identity Card
<div className="flex flex-col sm:flex-row items-start gap-4 sm:gap-5">
  {/* ... ícone + título ... */}
  {/* Coluna de datas - mover para baixo em mobile */}
  <div className="flex flex-row sm:flex-col gap-4 sm:gap-2 shrink-0 text-sm sm:ml-auto">
```

---

### Prioridade 3 — Alta (impacto nas abas de detalhes de empresa)

**Problema:** `TabsList grid-cols-5` comprime 5 abas em 320 px, tornando o texto ilegível.

**Páginas afetadas:** `/hr/companies/[id]` e `/hr/companies/[id]/edit`.

**Correção:**

```tsx
// Opção 1: Scroll horizontal nas abas
<TabsList className="flex w-full overflow-x-auto mb-4 p-2 h-auto">
  <TabsTrigger className="flex-none px-4 py-2" ...>

// Opção 2: Reduzir para 2 colunas em mobile
<TabsList className="grid w-full grid-cols-3 sm:grid-cols-5 mb-4 p-2 h-12">
```

---

### Prioridade 4 — Alta (impacto no componente Header compartilhado)

**Problema:** `Header` usa `flex-row` fixo sem colapsar para `flex-col` em mobile. Com título longo + múltiplos botões de ação, o título é comprimido.

**Arquivo:** `src/components/layout/header.tsx` (linha 72).

**Correção:**

```tsx
<div className={cn(
  'flex flex-col gap-3 sm:flex-row sm:items-center sm:justify-between sm:gap-6',
  className
)}>
```

---

### Prioridade 5 — Média (touch targets e SelectionToolbar)

**Problema:** Botões de ação rápida nos cards usam `size="sm"` (32 px de altura), abaixo do mínimo de 44 px recomendado para touch. O `SelectionToolbar` pode transbordar em telas estreitas.

**Correção:**

- Aumentar botões de ação inline para `min-h-[44px]` em mobile:
  ```tsx
  className = 'flex-1 text-xs min-h-[44px] sm:min-h-0';
  ```
- Adicionar `max-w-[calc(100vw-2rem)] overflow-x-hidden` ao wrapper do `SelectionToolbar`.

## Observações Gerais

- O sistema OpenSea OS com `EntityGrid`, `EntityCard`, `PageLayout` e `PageActionBar` oferece uma base responsiva sólida. A maioria dos problemas encontrados são localizados nas páginas específicas, não nos componentes core.
- O padrão de `hidden sm:inline` nos textos de botões (`Header`) e ícones de abas (`hidden sm:inline`) demonstra consciência sobre mobile, mas não foi aplicado de forma consistente em todos os pontos.
- A ausência de componentes `Sheet` (bottom sheet) para ações em mobile é uma limitação de UX, não funcional. Para o MVP, os dialogs centralizados são aceitáveis.
- O módulo não usa tabelas HTML — todos os dados são apresentados em cards, o que é intrinsecamente mais responsivo.
- Os gráficos Recharts com `ResponsiveContainer` são a abordagem correta para responsividade de gráficos.
