# Cores

A paleta do OpenSea está estruturada em três camadas (ver `src/app/globals.css`):

1. **Primitivos** — `--os-{color}-{shade}` (gray, slate, blue, rose, green, orange,
   purple). Não use diretamente em componentes.
2. **Semânticos** — `--color-{role}` (background, foreground, primary, destructive,
   etc.). É o que componentes consomem.
3. **Componente** — `--btn-*`, `--card-*`, `--badge-*`, etc. Refinam o token
   semântico para um componente específico.

Esta página foca nos tokens semânticos — é o nível em que telas novas são
implementadas.

Cada token é mostrado em **light** + **dark** lado a lado, com o valor real
extraído de `:root` e `.dark` em `globals.css`.

## Destrutivo é Rose, não Red

Regra do projeto (CLAUDE.md §9.1): **toda ação destrutiva usa `rose`, nunca `red`**.
Isso vale para botões, alertas, badges, ícones de exclusão, animação
`error-highlight` e bordas de erro.

<Grid min={220}>
  <ColorSwatch
    name="rose-500 (uso)"
    value="rgb(244 63 94)"
    textColor="light"
    description="Excluir / revogar / cancelar"
  />
  <ColorSwatch
    name="red-500 (NÃO usar)"
    value="rgb(239 68 68)"
    textColor="light"
    description="Bootstrap-style — proibido"
  />
</Grid>

**Por quê rose?** Tom mais quente, integra melhor à UI clara/escura do OpenSea e
evita o vermelho "agressivo" estilo Bootstrap. O `red-500` permanece apenas como
_primitive_ (`--os-red-500`) por razões de compatibilidade — não consuma direto.

## Status semantic

Cada estado tem um par light/dark de tons de superfície (`-50` no light, `-500/8`
no dark) e texto (`-700` no light, `-300` no dark). Combine com o componente
`<Badge>` — veja [Badge story](?path=/docs/ui-badge--default).

## Chips dual-theme

Padrão recorrente em CLAUDE.md §9: para chips/badges semânticos, use **classes
diferentes em light e dark** — light com fundo `-50` opaco e texto `-700`; dark
com fundo `-500/8` translúcido e texto `-300`. Isso garante contraste WCAG AA em
ambos os temas sem brilho excessivo no modo escuro.
